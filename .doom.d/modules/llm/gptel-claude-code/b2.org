This file tests: MCP tool called BY Claude Code, permission prompt, subagent inline output.

Prerequisites:

2. Set debug logging: =(setq gptel-log-level 'debug)=

This tests that Claude Code can discover and call Emacs MCP tools
during a query. The MCP config with session-aware URL should route
the tool call back to this buffer's session.

- [ ] No MCP routing errors in =*Messages*= or =*Warnings*=

* Test 5: Permission prompt triggered
This test requires changing permission mode from =bypassPermissions= to =default=.

** Step 0: Change permission mode
Evaluate this elisp BEFORE sending the test message:

#+begin_src elisp :results silent
(setf (gptel-claude-code-permission-mode
       (alist-get "Claude Code" gptel--known-backends nil nil #'equal))
      "default")
#+end_src

** Step 1: Trigger permission prompt
Send this - Claude Code should ask for permission to run the bash command,
which triggers the =permission_prompt= MCP tool and shows a popup buffer.

@user: Run this bash command: echo "hello from permission test"

** Step 2: Respond to permission prompt
When the =*gptel-permission*= buffer pops up:
1. Read the permission request
2. Press =a= to allow (or =d= to deny)
3. The response should continue after your choice

** Step 3: Restore bypass mode
After testing, restore bypass mode:

#+begin_src elisp :results silent
(setf (gptel-claude-code-permission-mode
       (alist-get "Claude Code" gptel--known-backends nil nil #'equal))
      "bypassPermissions")
#+end_src

** Verification
- [ ] =*gptel-permission*= buffer appeared with permission request
- [ ] Buffer showed the action description and allow/deny options
- [ ] Pressing =a= allowed the action and returned result to Claude Code
- [ ] Claude Code continued and showed the bash output "hello from permission test"
- [ ] Pressing =d= (if tested separately) denied the action gracefully
- [ ] =*Messages*= shows =--permission-mode default=
- [ ] =*Messages*= shows =--permission-prompt-tool mcp__emacs__permission_prompt=
